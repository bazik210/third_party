3. 更新情報 トップに戻る

"__unaligned" pointers

今回のリリースでは、 「__unaligned」 キーワードへのサポートが追加されました。これはポインタ定義における型修飾子です。ポインタが __unaligned 修飾子で宣言されると、コンパイラでは、適切にアラインされていないデータのアドレスを当該ポインタが取るものと仮定されます。 __unaligned で宣言されたポインタを通じてデータへのアクセスが行われる場合、コンパイラでは、アラインメントエラーを引き起こすことなくデータの読み書きを行うために必要なコードが追加生成されます。この追加コードの使用に対してはパフォーマンスが犠牲となるため、なるべくデータが正しくアラインされるようにすることが最善となります。

Cell PPU プロセッサにおける浮動小数点型およびベクター データ型の読み書きでは、正しくアラインされていることが必須であり、それ以外の場合は例外が発生します。当該ハードウェアでは、不適切なアラインの整数型には問題なくアクセスが行われます。

この修飾子では、アドレス指定されるデータのアラインメントのみが説明されます。ポインタ自体はアラインされていると仮定されます。

たとえば、以下のコードでは意図的に不適切なアラインのアクセスが行われますが、 __unaligned キーワードを使用することにより、コードが問題なく実行されるようになります。 

float read (float __unaligned * f)
{
    return *f;
}

char x [5];
float f;

void foo (void)
{
    f = read (&x [1]);
}

Changes

#67280
Improve compatibility with GCC for the handling of typedefs where an __attribute__((aligned (x))) is applied to the typedefed name, rather than to the underlying type. These types do not have the padding added that would ensure that members of an arrays of this type would be correctly aligned, so the compiler will reject arrays of such types. (BZ68022)
#67041
Added support for using the function name pre-definitions (e.g. __FUNC__ ) in message pragmas. (BZ77911)
#66824
Added support for the __unaligned keyword to allow user code to explicitly indicate that a pointer may not have the correct alignment for its type. (BZ71980)
#63075
This release changes the way the compiler identifies 'system header files' for purposes of suppressing warnings in them, and for the generation of "makefile style" dependency information via the -MMD option. Prior to SNC 310.1, any include that used the "angle bracket" notation, such as: 
#include <filename>
was identified as a 'system header file', whereas any include using the "quoted" notation: 
#include "filename"
was identified as a 'non-system header file'. The new rule is that files that are found via implicitly inserted include directories (usually inside the cell SDK) are deemed to be system header files, except for any directories explicitly added on the command-line via the -I option, which was not already being implicitly inserted. For example: 
-I c:\usr\local\cell\...
The syntax of the include directive (i.e., <> versus "") is now irrelevant for classifying a file as a system header file. Note that warnings are suppressed for constructs in 'system header files' by default (see the -Xnosyswarn switch); therefore, new warnings may appear when moving to SNC 310.1. In particular, if a non-system header file (say "my_header.h") is included via: 
#include <my_header.h>
and "my_header.h" contains constructs that would normally result in a warning being produced, then prior to release 310.1, the warning would have been suppressed because "my_header.h" was viewed as a system header file. Since "my_header.h" will no longer be interpreted as a system header file, the warnings will now appear. (BZ75846)
#62821
A change has been made to the diagnostics that the compiler issues when an assignment is detected in an 'if' statement. The compiler now detects two different uses of such assignments: 
if (x = 3)  // issues diagnostic 'assign_const_where_compare_meant'
if (x = y)  // issues diagnostic 'assign_where_compare_meant'
(BZ77945)
#62780
Implemented a "tail duplication" transformation to improve runtime performance: When a block branches unconditionally to a very short block, the code of that short target block is duplicated at the end of the original block, thus saving a branch at runtime. (To minimize increases in code-size, this transformation is only enabled within loops.)
#62635
The compiler now issues remarks on declaration of entities with reserved identifiers (those containing double underscores or starting with an underscore followed by an upper case letter) in user code, as in: 
int __foo;
x.c(1,4): remark 1779: Identifier __foo  is non-standard
  int __foo;
Note that these are remarks, which are normally suppressed, but can be enabled by either -Xdiag=2 (enable all remarks) or via --diag_warning=1779. These remarks are suppressed in system header files irrespective of -Xnosyswarn control. (BZ57644)

SNC と GCC の重要な相違点

詳細および GCC から SNC へのコードの移植に関するアドバイスは、『SNC 移行ガイド』を参照してください。

C と C++ のリンク互換性

SNC では GCC と同じ C++ ABI が使用されています (IA64 C++ ABI に基づく)。C and C++ code built by SNC and GCC may be mixed freely.

C++ クラス レイアウト互換性と vtable ポインタの配置

SNC では GCC と同じ C++ ABI が使用されています (IA64 C++ ABI に基づく)。The layouts of classes and vtables produced by the two compilers are identical.

SDK パッチ、インクルード パス、ライブラリ

このリリースでは、SDK からのヘッダー ファイルのパッチ バージョンを使用する必要がなくなりました。

以下は、SNC で使用されるパスを含むデフォルトです。これらは、コンパイラ ドライバによって設定されますが、-nostdinc オプションを使用して抑制することもできます。相対パスが使用される場合、これらはコンパイラ実行ファイルに相対的となります。このため、ツールチェーンは SDK にインストールする必要があります。正しくインストールされると、ツール実行ファイルは「{cell sdk}\host-win32\sn\bin」ディレクトリに配置されます。

  • ../../../host-win32/sn/ppu/include,
  • ../../../host-win32/sn/ppu/include/sn,
  • ../../../host-win32/sn/common/include,
  • ../../../host-win32/sn/common/include/sn,
  • ../../../target/ppu/include,
  • ../../../target/common/include,
  • $(SN_PS3_PATH)/ppu/include,
  • $(SN_PS3_PATH)/ppu/include/sn,
  • $(SN_PS3_PATH)/common/include,
  • $(SN_PS3_PATH)/common/include/sn,

インストーラにより、他の SN 製品に対する SN_PS3_PATH 環境変数が設定されます。この変数を参照するインクルードディレクトリは、これらの製品と共に提供される追加ヘッダー ファイルの場所を指定するために使用されます。

リンク付けとライブラリ

SNC でコンパイルされたアプリケーションは、SN ランタイム ライブラリ、および SDK に付属する GCC 仕様のライブラリ (このランタイム コンポーネントは SDK で必要) とリンクされている必要があります。

デフォルトのリンカー スクリプト $(CELL_SDK)/target/ppu/lib/elf64_lv2_prx.sn が、ライブラリ検索パス、ライブラリおよび一般的なアプリケーションへのリンク付けが必要なその他のラインタイムコンポーネントのデフォルト設定に含まれるようになりました。C++ 例外サポートのあるリンク付けとないリンク付けについて別個の設定があります。デフォルトでは、例外のない方のライブラリやランタイムが使用されます。 例外処理のある方を選択するには、-exceptions スイッチをリンカーに渡します

デフォルトは、リンク スクリプトの下記のセクションで指定されています。

  • LIB_SEARCH_PATHS - ライブラリ検索パス
  • STANDARD_LIBRARIES - 標準ライブラリ
  • REQUIRED_FILES - 必須の C および C++ ランタイム コンポーネント

デフォルトの設定は、SDK に対してリンクするアプリケーションの大部分で十分なはずです。追加的なライブラリや検索パスを、リンカー コマンドラインに追加することができます

以下は、SNC ランタイム対応ライブラリです。

  • libsn.a (ユーティリティとデバッグに対応)
  • libsnc.a (C ランタイムに対応)

サンプル

SnMathLib は、汎用および Altivec 対応の C++ 演算ライブラリのサンプルです。

  • host-win32\sn\common\include\sn\SnMathLibDef.h and SnMathLibGeneric.h
  • host-win32\sn\ppu\include\sn\SnMathLibPPU.h

制限

警告

  • SNC では、GCC スタイル制約でのインライン asm 使用に対応していません (未加工のアセンブリ言語には対応)。オプティマイザとのより緊密な統合が行えるため、低レベルのコードに対しては組込み関数ベースのアプローチが推奨されます。組み込み関数でサポートされないコード例がある場合は、当社までご連絡ください。
